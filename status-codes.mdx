## HTTP Status Codes

Klivopay uses standard HTTP status codes to indicate the success or failure of API requests.

### Success Codes

<ResponseField name="200 OK" type="success">
  Request succeeded
</ResponseField>

<ResponseField name="201 Created" type="success">
  Resource created successfully
</ResponseField>

### Client Error Codes

<ResponseField name="400 Bad Request" type="error">
  The request was invalid or cannot be served
</ResponseField>

<ResponseField name="401 Unauthorized" type="error">
  Authentication failed or API token is invalid
</ResponseField>

<ResponseField name="404 Not Found" type="error">
  The requested resource does not exist
</ResponseField>

<ResponseField name="422 Unprocessable Entity" type="error">
  Validation error - check the response for details
</ResponseField>

<ResponseField name="429 Too Many Requests" type="error">
  Rate limit exceeded
</ResponseField>

### Server Error Codes

<ResponseField name="500 Internal Server Error" type="error">
  Something went wrong on Klivopay's end
</ResponseField>

<ResponseField name="503 Service Unavailable" type="error">
  Service temporarily unavailable
</ResponseField>

## Error Response Format

All error responses follow this format:

```json
{
  "error": "Error type",
  "message": "Detailed error message",
  "code": "ERROR_CODE",
  "details": {
    // Additional error details
  }
}
```

### Example Error Responses

#### 400 Bad Request

```json
{
  "error": "Bad Request",
  "message": "Invalid payment method",
  "code": "INVALID_PAYMENT_METHOD"
}
```

#### 401 Unauthorized

```json
{
  "error": "Unauthorized",
  "message": "Invalid API token",
  "code": "INVALID_TOKEN"
}
```

#### 422 Validation Error

```json
{
  "error": "Validation Error",
  "message": "The given data was invalid",
  "code": "VALIDATION_ERROR",
  "details": {
    "customer.email": ["The email field is required"],
    "amount": ["The amount must be greater than 0"]
  }
}
```

## Handling Errors

### Best Practices

<AccordionGroup>
  <Accordion title="Always Check Status Codes">
    Don't assume success - always check the HTTP status code before processing the response.
  </Accordion>

  <Accordion title="Log Error Details">
    Log the complete error response for debugging, including the `code` and `details` fields.
  </Accordion>

  <Accordion title="Implement Retry Logic">
    For 5xx errors and rate limits (429), implement exponential backoff retry logic.
  </Accordion>

  <Accordion title="Handle Validation Errors">
    Parse the `details` object to show specific field errors to users.
  </Accordion>

  <Accordion title="Use Timeouts">
    Set appropriate timeouts (30 seconds recommended) to handle slow responses.
  </Accordion>
</AccordionGroup>

### Example Error Handling

```javascript
try {
  const response = await createTransaction(data);
  console.log('Transaction created:', response);
} catch (error) {
  if (error.response) {
    // Server responded with error
    const { status, data } = error.response;
    
    switch (status) {
      case 400:
        console.error('Bad request:', data.message);
        break;
      case 401:
        console.error('Authentication failed');
        // Refresh or request new API token
        break;
      case 422:
        console.error('Validation errors:', data.details);
        // Show field-specific errors to user
        break;
      case 429:
        console.error('Rate limit exceeded, retrying...');
        // Implement retry with backoff
        break;
      case 500:
        console.error('Server error, please try again later');
        break;
      default:
        console.error('Unexpected error:', status, data);
    }
  } else {
    // Network or timeout error
    console.error('Network error:', error.message);
  }
}
```

## Rate Limiting

<Warning>
  While specific rate limits are not documented, implement reasonable throttling in your application to avoid hitting limits.
</Warning>

### Recommendations

- Don't make more than 10 requests per second
- Implement exponential backoff for retries
- Cache responses when possible
- Use batch operations when available
